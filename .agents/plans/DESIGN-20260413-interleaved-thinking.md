# Design: Interleaved Thinking as Default

**Date**: 2026-07-13
**Status**: Draft — awaiting approval

## Problem Statement

Interleaved thinking — the model reasoning *between* tool calls rather than only before the first one — dramatically improves agentic reliability. MiniMax benchmarks show +36% on Tau² and +40% on BrowseComp when thinking state is preserved. Anthropic, OpenAI, and MiniMax all now support it, but our codebase has three gaps:

1. **Ordering bug in `convertMessages`** — Assistant message parts are grouped by type (all thinking, then all text, then all tool_calls), destroying the model's interleaved output sequence.
2. **No interleaved-thinking beta header** — Anthropic and Bedrock require `interleaved-thinking-2025-05-14` to enable this mode. The Claude provider already sends it; the direct Anthropic and MiniMax providers don't.
3. **Bedrock doesn't round-trip thinking blocks** — The Bedrock message builder ignores `PartTypeThinking` entirely on assistant messages.

The fix has one core principle: **preserve the exact part order the model emitted**, across all providers.

## Scope

**In scope:**
- Fix part ordering in all provider message builders (Anthropic, Bedrock, OpenRouter)
- Add the interleaved-thinking beta header (Anthropic direct, Bedrock, MiniMax)
- Bedrock: serialize thinking parts on assistant messages
- Ensure all of this is the default for models that support it (no opt-in flag)

**Out of scope:**
- OpenAI Responses API reasoning item round-trip (different protocol — reasoning items are opaque/encrypted, not `thinking` blocks; separate future work)
- Thinking block clearing / context management betas
- Adaptive thinking for Bedrock (Bedrock doesn't support it yet)

## Impact of Uncommitted Changes

There's an in-flight refactor (`request.go`, all providers) that consolidates `ThinkingEffort` + `OutputEffort` into `Effort` + `ThinkingMode`. This design is written against the **uncommitted state**:

| Old API | New API (uncommitted) | Rationale |
|---|---|---|
| `ThinkingEffort` depth aspect (minimal→xhigh) | `Effort` (low/medium/high/max) | Merged with `OutputEffort` into one knob |
| `ThinkingEffort` on/off aspect (none vs rest) | `ThinkingMode` (auto/on/off) | Separated "whether to think" from "how hard" |
| `OutputEffort` (low/medium/high/max) | `Effort` (low/medium/high/max) | Absorbed into unified Effort |

All code references below use the new types. The in-flight refactor doesn't touch `convertMessages`, the beta header, or Bedrock thinking block serialization — so there are no conflicts.

### What's broken or missing

#### 1. Ordering bug — `convertMessages` (Anthropic, line 154-169)

```go
// Current: groups by type → destroys interleaved order
case llm.RoleAssistant:
    var blocks []MessageContentBlock
    // 1. ALL thinking first
    for _, t := range m.Parts.ByType(msg.PartTypeThinking) { ... }
    // 2. ALL text next
    for _, p := range m.Parts.ByType(msg.PartTypeText) { ... }
    // 3. ALL tool calls last
    for _, p := range m.Parts.ByType(msg.PartTypeToolCall) { ... }
```

With interleaved thinking, the model emits blocks like:
```
thinking(0) → tool_use(1) → thinking(2) → text(3) → tool_use(4)
```

Grouping by type reorders to:
```
thinking(0) → thinking(2) → text(3) → tool_use(1) → tool_use(4)
```

This corrupts the reasoning chain — the model's thoughts about tool results appear *before* the tool calls they reference. More critically, the Anthropic and Bedrock APIs enforce that "the entire sequence of consecutive thinking blocks must match the outputs generated by the model during the original request; you cannot rearrange or modify the sequence of these blocks" ([Bedrock docs](https://docs.aws.amazon.com/bedrock/latest/userguide/claude-messages-extended-thinking.html), [OpenRouter docs](https://openrouter.ai/docs/guides/best-practices/reasoning-tokens)). Reordering violates this constraint and will cause API errors.

> **Note**: The older constraint that "assistant messages must start with a thinking block" is *relaxed* by the interleaved thinking beta — tool_use blocks may now precede thinking blocks. But the sequence-must-match constraint remains strict.

#### 2. Missing beta header (Anthropic direct + MiniMax)

The **Claude provider** already sends the header:
```go
// provider/anthropic/claude/provider.go, line 25:
claudeBeta = "claude-code-20250219,oauth-2025-04-20,interleaved-thinking-2025-05-14,..."
```

But the **direct Anthropic provider** (`provider/anthropic/anthropic.go`) and **MiniMax** don't send any `anthropic-beta` header. Bedrock needs it in `additionalModelRequestFields`.

#### 3. Bedrock doesn't round-trip thinking blocks

```go
// Current Bedrock assistant message builder — no thinking parts
case msg.RoleAssistant:
    var content []types.ContentBlock
    if m.Text() != "" {
        content = append(content, &types.ContentBlockMemberText{Value: m.Text()})
    }
    for _, tc := range m.ToolCalls() { ... }
```

Thinking parts are silently dropped. The Bedrock SDK **does** support `ContentBlockMemberReasoningContent` as an input content block (verified: it implements `isContentBlock()`), with `ReasoningTextBlock{Text, Signature}`. No workaround needed.

## Design

### Principle: iterate `Parts` in order

All message builders must iterate `m.Parts` in sequence, switching on `part.Type`, rather than grouping by type via `ByType()`. This preserves interleaved order.

### Change 1: Fix `convertMessages` in `provider/anthropic/message.go`

**Before** (lines 154-169):
```go
case llm.RoleAssistant:
    var blocks []MessageContentBlock
    for _, t := range m.Parts.ByType(msg.PartTypeThinking) { ... }
    for _, p := range m.Parts.ByType(msg.PartTypeText) { ... }
    for _, p := range m.Parts.ByType(msg.PartTypeToolCall) { ... }
```

**After**:
```go
case llm.RoleAssistant:
    var blocks []MessageContentBlock
    for _, p := range m.Parts {
        switch p.Type {
        case msg.PartTypeThinking:
            blocks = append(blocks, Thinking(p.Thinking.Text, p.Thinking.Signature))
        case msg.PartTypeText:
            blocks = append(blocks, Text(p.Text))
        case msg.PartTypeToolCall:
            tc := p.ToolCall
            tub := ToolUse(tc.ID, tc.Name, tc.Args)
            blocks = append(blocks, &tub)
        }
    }
```

Same blocks, same logic, just iterated in the model's emission order.

**Impact**: Anthropic direct API and MiniMax (both call `convertMessages` via `anthropic.BuildRequest`).

### Change 2: Add interleaved-thinking beta header

**`provider/anthropic/anthropic.go`** — in `newAPIRequest` (line 132):
```go
req.Header.Set("Anthropic-Beta", "interleaved-thinking-2025-05-14")
```

**`provider/minimax/minimax.go`** — in `newAPIRequest` (after line 153):
```go
req.Header.Set("Anthropic-Beta", "interleaved-thinking-2025-05-14")
```

**`provider/bedrock/bedrock.go`** — new block alongside `topK`, `output_schema`, and `reasoning_config`:
```go
// Always enable interleaved thinking beta for Claude models.
// Harmless no-op for models that don't support it.
if isClaudeModel(opts.Model) {
    if additionalFields == nil {
        additionalFields = make(map[string]any)
    }
    additionalFields["anthropic_beta"] = []string{"interleaved-thinking-2025-05-14"}
}
```

(Confirmed via AWS Strands SDK blog: array of strings in `additionalModelRequestFields` is the correct mechanism for Bedrock. Uses the same `nil` guard pattern as the existing `topK` and `reasoning_config` blocks. Scoped to Claude models via `isClaudeModel()` since non-Anthropic Bedrock models don't recognize this field.)

**No change needed** for `provider/anthropic/claude/provider.go` — it already sends the header at line 25/181.

The header is always sent. Models that don't support it ignore it. The API version header controls compatibility, not the beta header.

### Change 3: Fix Bedrock assistant message builder

**`provider/bedrock/bedrock.go`**, the `RoleAssistant` case (lines 406-430):

**Before**:
```go
case msg.RoleAssistant:
    var content []types.ContentBlock
    if m.Text() != "" {
        content = append(content, &types.ContentBlockMemberText{Value: m.Text()})
    }
    for _, tc := range m.ToolCalls() { ... }
```

**After** — iterate parts in order, including thinking:
```go
case msg.RoleAssistant:
    var content []types.ContentBlock
    for _, p := range m.Parts {
        switch p.Type {
        case msg.PartTypeText:
            if p.Text != "" {
                content = append(content, &types.ContentBlockMemberText{Value: p.Text})
            }
        case msg.PartTypeThinking:
            content = append(content, &types.ContentBlockMemberReasoningContent{
                Value: &types.ReasoningContentBlockMemberReasoningText{
                    Value: types.ReasoningTextBlock{
                        Text:      aws.String(p.Thinking.Text),
                        Signature: aws.String(p.Thinking.Signature),
                    },
                },
            })
        case msg.PartTypeToolCall:
            tc := p.ToolCall
            inputDoc, err := toDocument(tc.Args)
            if err != nil {
                return nil, fmt.Errorf("marshal tool arguments: %w", err)
            }
            content = append(content, &types.ContentBlockMemberToolUse{
                Value: types.ToolUseBlock{
                    ToolUseId: aws.String(tc.ID),
                    Name:      aws.String(tc.Name),
                    Input:     inputDoc,
                },
            })
        }
    }
```

**SDK verification**: `ContentBlockMemberReasoningContent` implements `isContentBlock()` (confirmed in `aws-sdk-go-v2/service/bedrockruntime@v1.50.0/types/types.go`). `ReasoningTextBlock` has `Text *string` and `Signature *string` fields. This is a first-class SDK type, not a workaround.

### Change 4: Fix OpenRouter message builder

The OpenRouter builder also groups thinking parts separately before tool calls. Same fix — iterate `m.Parts` in order:

**Before** (openrouter.go ~line 319-342):
```go
case msg.RoleAssistant:
    mp := messagePayload{Role: "assistant", Content: m.Text()}
    thinkingParts := m.Parts.ByType(msg.PartTypeThinking)
    if len(thinkingParts) > 0 {
        // ... all thinking first ...
    }
    for _, tc := range m.ToolCalls() { ... }
```

**After**:
```go
case msg.RoleAssistant:
    mp := messagePayload{Role: "assistant", Content: m.Text()}
    var reasoningText strings.Builder
    for _, p := range m.Parts {
        switch p.Type {
        case msg.PartTypeThinking:
            if p.Thinking == nil { continue }
            reasoningText.WriteString(p.Thinking.Text)
            mp.ReasoningDetails = append(mp.ReasoningDetails, reasoningDetailInput{
                Type:      "reasoning.text",
                Text:      p.Thinking.Text,
                Signature: p.Thinking.Signature,
            })
        case msg.PartTypeToolCall:
            argsJSON, _ := json.Marshal(p.ToolCall.Args)
            mp.ToolCalls = append(mp.ToolCalls, toolCallItem{
                ID:   p.ToolCall.ID,
                Type: "function",
                Function: functionCall{Name: p.ToolCall.Name, Arguments: string(argsJSON)},
            })
        }
    }
    if reasoningText.Len() > 0 {
        mp.Reasoning = reasoningText.String()
    }
```

Note: OpenRouter's wire format has separate `reasoning_details` and `tool_calls` arrays (they don't interleave at the JSON level), but the order within `reasoning_details` must match the model's emission order for correct round-trip. OpenRouter handles the interleaving server-side.

## No API changes required

- `ThinkingMode` stays as-is: `ThinkingAuto` (default), `ThinkingOn`, `ThinkingOff`
- No new fields on `Request`, `Message`, or `Part`
- Interleaved thinking is not a separate mode — it's an API capability activated by the beta header. Once activated, the model *may* produce thinking blocks between tool calls depending on the task.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| MiniMax may not recognize `Anthropic-Beta` header | Harmless — MiniMax natively enables interleaved thinking for M2+ models ([source](https://simonwillison.net/2025/Nov/3/minimax/)); header is ignored |
| Old Claude 3.7 Sonnet doesn't support interleaved | Header is a no-op for unsupported models; no error |
| Bedrock `reasoning_config` field name vs `thinking` | Both work in `additionalModelRequestFields`. The Bedrock docs and Strands blog use `thinking`; our code and community examples (e.g. [Medium](https://seobs.medium.com/using-sonnet-3-7-reasoning-with-the-converse-api-on-aws-bedrock-6c838c685ad9)) use `reasoning_config`. No change needed. |
| `anthropic_beta` conflicts with other entries | Bedrock: array of strings, so append-safe. Anthropic direct: we set one header value; no existing beta headers to conflict with. |
